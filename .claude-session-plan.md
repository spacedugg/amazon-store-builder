## Step-by-Step-Fahrplan

### ✅ Step 1: Cleanup — commit aaafdb9

- Legacy `src/generationPipeline.js` gelöscht (363 Zeilen)
- 23-Brand-Aggregate `_summary.json` + `_aggregated_analysis.json` gelöscht
- `COMPLEXITY_LEVELS` komplett entfernt (constants.js, storeBuilder.js, App.jsx, GenerateModal.jsx, index.css, i18n.js)
- `analyzeBrandVoice` + `checkAsinCompleteness` migriert zu contentPipeline.js
- `analyzeBrandVoice` erweitert um: voiceFingerprint, sentencePatterns, vocabulary.signatureWords, vocabulary.avoidedWords, do[], dont[], visualToneCues[], ctaStyle, typicalPhrases
- Kategorie-Name-als-Slug-Hard-Rule gesoftet

### 🔄 Step 2: Brand Intelligence Module (IN ARBEIT)

**Ziel:** EIN strukturiertes Brand-Intelligence-Objekt, das voice + visual + content-inventory + reuse-flags bündelt und durch die gesamte Pipeline fließt.

**Getan:**
- `buildBrandIntelligence()` und `formatBrandIntelligenceForPrompt()` in contentPipeline.js hinzugefügt (reine Packaging-Funktion, kein API-Call)

**Offen:**
- Signatur-Anpassung `synthesizeBrandProfile`, `planPages`, `generateOnePage` — nehmen `brandIntelligence` als Parameter
- Prompts dieser Funktionen mit `formatBrandIntelligenceForPrompt(bi)` anreichern
- In App.jsx neue Phase 2.7 zwischen PHASE 2.5 (Brand Voice) und PHASE 3 (Brand Profile) einbauen:
  `brandIntelligence = buildBrandIntelligence({...all collected data...})`
- Alle Call-Sites von synthesizeBrandProfile / planPages / generateOnePage auf neue Signatur umstellen
- `storeData.brandIntelligence = brandIntelligence` persistieren
- Log-Ausgabe im Pipeline-Panel: "Intelligence: X USPs, Y Colors, Z Reuse-Flags"

### 📋 Step 3: Blueprint Matching

**Ziel:** Die 20 Deep-Analysen als strukturelle Skelette nutzen, nicht als freien Kontext-Text.

**Aktionen:**
- `loadStoreKnowledge()` und `formatStoreKnowledge()` prüfen/erweitern
- Neue Funktion `matchBlueprints(brandIntelligence, categories, productCount, lang)` in contentPipeline.js:
  - Nimmt brandIntelligence (voice.toneDescriptors + visual.visualMood + reuseFlags)
  - Vergleicht mit den 20 Analysen anhand: Tone-Affinität, Product-Count-Band, Visual-Mood-Match, Struktur-Komplexität
  - Gibt Top 3-5 Blueprints zurück als `{ brand, pageCount, typicalPages, modulePatterns, cadence, insightSnippet }`
- `planPages` bekommt die Top-Blueprints als strukturelle Inspiration (nicht als Text-Dump)
- `generateOnePage` bekommt für die jeweilige Seite die passende Module-Pattern-Abfolge

### 📋 Step 4: Per-Page Realization Refactor

**Ziel:** `generateOnePage` soll konsumieren: voice + blueprint-skelett + content-inventory + reuse-flags. Keine hardcodierten Section-Regeln mehr — das Trio entscheidet.

**Aktionen:**
- `generateOnePage` Signatur: `(pagePlan, brandIntelligence, categories, productAnalyses, brand, lang, previousPages, matchedBlueprint)`
- Prompt-Umbau: weg von "AVAILABLE LAYOUTS" Litanei, hin zu "Blueprint zeigt dir für diese Seite folgende Module-Kadenz — pass sie an deine Marke an"
- `contentInventory.reusablePhrases` explizit in die Copy-Instruktion: "nutze diese Marken-Phrasen dort wo es passt"
- `reuseFlags.adoptExistingContent === true` → Prompt weist an, Existing-Store-Headlines bevorzugt zu übernehmen
- `enforceHomepageProductGridPlacement` etc. in storeBuilder.js überprüfen, ob noch gebraucht oder obsolet

### 📋 Step 5: Wizard-Extension (Einzelfall-Logik für Existing Store)

**Ziel:** Klares, separates Wizard-Feld für "Hast du schon einen Brand Store?" mit expliziter Inhalte-Übernahme-Option.

**Aktionen:**
- In `GenerateModal.jsx` (oder `GenerationWizard.jsx` — beide prüfen):
  - Neues Eingabefeld "Bestehender Amazon Brand Store (URL)" visuell prominent
  - Toggle-Gruppe "Modus": Optimieren | Neu konzipieren (existingStoreMode — ist schon da)
  - NEUE Checkbox "Inhalte übernehmen" (nur sichtbar wenn Modus=Optimieren) — wird als `params.adoptExistingContent` durchgereicht
  - NEUE Checkbox "Menüstruktur behalten" (existingStoreMode=Optimize) — keepMenuStructure existiert schon
- In `App.jsx` `params.adoptExistingContent` abgreifen und an `buildBrandIntelligence()` übergeben
- UI-Kontext-Hilfe: kurzer Erklärtext unter Checkbox

### 📋 Step 6: Post-Processing Voice Consistency Check

**Ziel:** Finaler Sanity-Check, der alle generierten Texte gegen das Voice-Playbook prüft und Abweichungen meldet/korrigiert.

**Aktionen:**
- Neue Funktion `voiceConsistencyCheck(pages, brandIntelligence, lang)` in contentPipeline.js
  - Sammelt alle textOverlay, ctaText, heroBannerTextOverlay
  - Ein API-Call an Claude: "prüfe ob diese Texte zum Voice-Fingerprint passen. Markiere Ausreißer. Gib korrigierte Varianten zurück."
  - Rückgabe: Array von `{ pageId, sectionIndex, tileIndex, field, original, issue, corrected }`
- In App.jsx als PHASE 7.5 zwischen Validation und Store-Build
- User sieht im Pipeline-Log: "Voice-Check: X Korrekturen angewendet"
- Im Store-Editor zeigen wir optional die Diff-Liste an

## Architektur-Insights (aus der Analyse vor der Umsetzung)

**Bestehende Pipeline sammelt schon viele Daten, führt sie aber NICHT strukturell zusammen:**
- `productCI` (analyzeBrandCI aus Amazon-Bildern) → liegt auf `enhancedWebsiteData.productCI`
- `existingStore` Analyse (crawlAndParseStore + Gemini) → als freier Text-Prefix an `referenceAnalysis` konkateniert
- `allWebsiteAnalyses` (scrapeWebsite pro Seite) → separates Array
- `pipelineBrandVoice` (analyzeBrandVoice) → eigenes Objekt
- Store-Knowledge (20 Analysen) → als Text über `formatStoreKnowledge` angehängt

**→ Step 2 macht genau das: alles in EIN Objekt `brandIntelligence` mergen, das durch alle Downstream-Calls fließt.**

**→ Step 3 verlegt Store-Knowledge von "Text-Dump" auf "strukturiertes Matching".**
